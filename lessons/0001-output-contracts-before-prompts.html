Lesson 0001

Output Contracts Before Prompts

Your first production habit: define the shape and acceptance criteria of an AI output before tuning a prompt. The model call should fit into typed application code, not leak uncertainty into the rest of the system.

The One Idea

A production LLM feature is not "prompt in, text out." It is a contract:

Task: classify and summarize a support ticket.

Schema: produce typed fields the app can validate.

Eval set: run representative tickets through the system and measure expected behavior.

Prompt wording comes after this. Without a schema and eval set, prompt changes are guesses.

A Small Contract

{
  "category": "hardware | software | billing | account | other",
  "priority": "low | normal | high | urgent",
  "summary": "short user-facing summary",
  "suggested_reply": "safe draft reply",
  "confidence": 0.0,
  "escalation_required": false
}

The application should reject missing fields, unknown categories, empty summaries, non-boolean escalation flags, and confidence values outside 0.0 to 1.0.

Use the legend, classify each ticket, then check your answers.

Project Slice

Now run the starter project. It implements the same idea with no API key: a typed schema, a baseline classifier, golden examples, and an evaluator.

cd projects/01-structured-support-triage
python3 -m unittest discover -s tests
PYTHONPATH=. python3 -m support_triage.cli --eval data/golden.jsonl

Once this is clear, the next step is replacing the baseline classifier with a real model call while keeping the schema and evals unchanged.
